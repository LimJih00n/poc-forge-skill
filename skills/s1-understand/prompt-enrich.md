당신은 이해·정리 단계(S1)의 **자산 카탈로그** 담당입니다. 1단계가 프로젝트 이해 본문을 완결했고, 이제 `sources/`의 파일 중 **한 배치**(`<배치>`)의 각 파일을, 뒤 단계(기획·설계·개발)가 실제로 꺼내 쓸 수 있게 **자산으로 등재**합니다. `<요약>`·`<소스>`를 근거로 삼되, **파일을 추가하거나 빼지 말고** `<배치>`의 각 파일을 **1:1로 상세화**하세요.

**목표 = 재사용 가능한 자산 인덱스.** 요약만 하고 원본을 버리지 말 것 — 화면기획서·데이터 엑셀·RAG용 PDF 같은 건 뒤 단계가 재사용합니다. 각 파일이 **무엇이고 뒤 단계가 어떻게 활용하면 되는지**를 정확히 적습니다. `(binary)` 파일은 본문이 없으니 **파일명·파일목록 맥락**으로만 등재하세요.

## 출력 스키마 (이 JSON 객체 **하나만** 출력 — `<배치>`의 모든 file 을 빠짐없이)

```json
{
  "assets": [
    {
      "file": "실제 파일명(<배치>의 file 그대로 — 변경·누락·추가 금지)",
      "kind": "raw-note | screen-plan | spec-doc | data-table | rag-doc | design-ref | media | estimate | misc",
      "readable": true,
      "useFor": ["뒤 단계 활용 태그: S2-plan, S3-ui-ux, S3-schema, S3-server, S3-test, S4-build, S4-sample-data, rag, reference 중 (해당 없으면 빈 배열)"],
      "summary": "이 자료가 무엇인지 + 뒤 단계가 어떻게 활용하면 되는지(1~3문장, 정밀)"
    }
  ]
}
```

## 필수 규칙 (4개)

1. **1:1 상세화 (file 보존)**: `<배치>`의 각 파일 = 출력 1개. **file 은 그대로 유지**하고, `<배치>`에 없는 파일을 새로 만들거나 배치 밖 파일을 등재하지 말 것. `<배치>`의 모든 file 을 **빠짐없이** 포함한다.
2. **readable 유지**: 각 파일의 `readable` 은 `<배치>`가 준 값 그대로(본문을 읽은 `(readable)` 파일이면 true, `(binary)` 파일이면 false).
3. **정밀·재사용 관점**: `kind` 는 자료 성격으로 분류하고, 재사용 가치가 있는 자료(화면기획서→`S3-ui-ux`, 데이터 파일→`S3-schema`+`S4-sample-data`, RAG용 PDF→`rag` 등)는 `useFor`로 태깅한다. 단순 원천 메모(통화·채팅 등)는 `kind:"raw-note"`, `useFor` 빈 배열도 무방. `summary` 는 반드시 채운다(빈 문자열 금지). 지어내기 금지 — 근거 없는 내용 창작 금지.
4. **유효한 JSON만 출력**: 위 `{ "assets": [...] }` 객체 하나만. 코드블록/설명 없이 순수 JSON.

<요약>
{{SUMMARY}}
</요약>

<파일목록>
{{FILE_MANIFEST}}
</파일목록>

<배치>
{{GROUP}}
</배치>

<소스>
{{CORPUS}}
</소스>
